# P6.2 — Goals, holes, unification

**Size:** L · **Depends on:** P6.1a · **The layer that does not exist**

Design: [`../design/03-architecture.md`](../design/03-architecture.md) §4.

## Goal

A representation of **not yet knowing**: a goal as data, forkable, with tracked
holes and enough unification to be goal-directed.

## Why this is the hard phase

Reconstruction has never needed a hole — a certificate *dictates* the proof, so
nothing ever represents an unknown (`reconstruct.rs:7921`: it "otherwise builds
closed terms"). Its control flow runs **backward from the answer**; a goal layer's
runs **forward from the question**. This phase is that inversion, and it is
greenfield.

`ExprNode` has 9 variants and **no `MVar`** (`expr.rs:80-98`).

## T6.2.4 — delayed assignment, in detail

**Get this wrong and `intro` is wrong, and everything above it is wrong.** It is
the most important thing to lift from Lean and the least obvious, and it is a
**type-theoretic necessity, not a syntactic convenience** — it survives deleting
the parser.

The problem: you cannot abstract a binder over a hole whose local context
*contains* that binder. The result is ill-formed. Lean's answer:

```
?m := ?n x        -- assign the mvar to an APPLICATION of a fresh one
```

Lean's own words (`Lean/MetavarContext.lean`): *"delayed assignment + application …
to implement a delayed substitution."* **Every `intro` and every `induction` hits
this.**

Reference: [`../research/12-elaboration-egraphs-fmf.md`](../research/12-elaboration-egraphs-fmf.md) §1.

## T6.2.5 — the depth invariant, in detail

> **Level *N+1* must be fully assigned before returning to level *N*.**

This is what stops a nested `simp` call from silently solving a **sibling** goal.

And **mvar kinds** carry the same weight: `SyntheticOpaque` is Lean's name for
*"this is an unsolved subgoal — do not let unification close it by accident."*
Adopt the concept and the name.

**Explicit metavariable coupling** is the sleeper: when two goals share a hole, an
agent must be able to ask *what else does solving this constrain?* Cheap while
designing the representation; **effectively unrecoverable afterwards**.

## What to skip, and why

From note 12 — these exist to disambiguate *what a user typed*, and API-built goals
are unambiguous:

| Skip | Because |
|---|---|
| **Hygiene** | It exists only to prevent capture in *user-written textual macros*. |
| **`do` notation, quotations** | Surface syntax. |
| **Coercion insertion** | Disambiguates a typed expression. |
| **Ad hoc overloading** | Same. |

Ullrich's thesis is literally titled *An Extensible Theorem Proving Frontend* — **a
thesis about the part we skip.** What survives *is* this phase.

## Phase exit criteria

- An agent obtains a goal, forks it, attempts it, observes a typed result — **with
  nothing proved yet**.
- Identical goals serialize identically.
- The unification fragment is published and declines outside it.
- **No hole can become a theorem.**

## The verification boundary

- **Newly in the TCB:** nothing, if T6.2.1 holds the line. The obligation type lives
  outside the kernel; only the checker's output is admitted.
- **Watch:** if `MVar` enters `ExprNode`, the TCB grows and every kernel traversal
  is newly in scope. That is a re-decision, not an implementation detail.
