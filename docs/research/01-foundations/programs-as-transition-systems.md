# Programs As Transition Systems

Status: draft
Last updated: 2026-06-10

## Purpose

Connect symbolic execution and verification to general transition-system reasoning.

## Scope

In scope:

- Program states, transition relations, bounded reachability, and witnesses.

Out of scope:

- Complete operational semantics for any specific ISA or language.

## Core Claims

- A program can be modeled as a transition system.
- A bounded execution path can be encoded as a finite conjunction of transition
  constraints.
- A reachability question is satisfiability of an existential formula over initial
  inputs and intermediate states.
- A found model is not enough for high confidence; it should be replayed in the
  original semantics or a trusted executable model.

## Basic Model

```text
State = registers x memory x pc x environment

Init(input, state_0)
Step(state_0, state_1)
Step(state_1, state_2)
...
Goal(state_k)
```

Bounded reachability:

```text
exists input, state_0..state_k.
  Init(input, state_0)
  and Step(state_0, state_1)
  and ...
  and Goal(state_k)
```

## Design Implications

- Axeyum should not hard-code "bug finding" as the only transition-system use case.
- APIs should support formulas generated by compilers, model checkers, protocol
  analyzers, binary analyzers, and math tools.
- Models should preserve named user variables and symbolic byte ranges.

## Risks

- State explosion is inherent; the solver stack cannot compensate for a poor
  exploration strategy.
- Memory and environment modeling often dominate pure expression solving.
