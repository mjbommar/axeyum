# P5.5 — A real external target, measured

## Goal

Point the front end at a **real, external Rust systems project** and commit a
measured result: a module verified (with evidence) or a real bug found and
reproduced — under the same scoreboard discipline as every other measured
claim (DISAGREE=0, committed baselines, wall-times, no seeding).

## Why / leverage

The standing course-correction applies to Track 5 doubly: **measure, don't
seed**. Fixtures we wrote can only prove the machinery works on what we chose
to write. One verified module of someone else's kernel — or one reproduced bug
— is the difference between a demo and a capability, and it is what makes the
positioning claim ("Verus-but-checkable / Hyperkernel-but-Rust") defensible in
public.

## Dependencies

- P5.1 (must reflect real build artifacts, not curated fixtures).
- At least one of P5.2 (if the module has internal calls) / P5.3 (if the
  obligation is memory/2-safety/FSM-shaped).
- Target selection should happen **early** (during P5.1) because it fixes the
  fragment-coverage priorities.

## Candidate targets (from the 2026-07-06 landscape survey; pick by smallest viable)

| Candidate | Why | Risk |
|---|---|---|
| **Maestro** (Linux-compatible Rust kernel, early-stage) | small, monolithic, plausibly bug-rich; a found+reproduced panic is a headline | young codebase churns; MIR fixture drift |
| **Hubris / Tock components** (production embedded Rust) | small, real, safety-critical, stable | teams may have stronger invariants already; fewer easy finds |
| **Asterinas OSTD slice** (the vostd/Verus target) | direct, benchmarkable head-to-head vs the Verus approach on the *same module* (page tables) | largest fragment demands (memory model); highest bar |
| **A `rust-sel4` userspace task** | composes with the seL4 story directly (QSOE-style) | smaller audience; toolchain setup |

Selection is recorded as a short decision note in this folder (not a full ADR
unless it forces new public surface).

T5.5.1 selects Maestro's three scalar device-number encode/decode functions at
an exact upstream revision. The [selection note](P5.5-target-selection-maestro-device-id.md)
records the candidate comparison, owning-build feasibility observation,
universal obligation shape, external-capture gaps, and the no-vendoring license
gate. No external artifact or result is admitted by selection alone.

ADR-0326 closes that owning-build route negative. The replacement
[selection note](P5.5-target-selection-tock-log2.md) chooses Tock's public
32/64-bit integer-log helpers at exact revision `ac5d597d`. ADR-0327 accepts
their checked LLVM-22 `range`/`ctlz` prerequisite. ADR-0328 now freezes the
zero-row authenticated build/extraction protocol. Its producer, ten focused
mutation/cleanup tests, and exact registration were pushed before invocation;
v1 then closes at the locked-offline cache preflight because `ghash 0.4.4` is
absent. Zero builds/target bytes exist. A dedicated-cache v2 requires a fresh
zero-result decision before preparation or builds. ADR-0329 freezes and runs
that input-only protocol, but preparation v2 closes before download because the
namespace omits the runtime target of the host resolver symlink. A fresh v3
must authenticate the minimal resolver file and a real DNS probe while keeping
the offline/inventory/no-compilation gates unchanged. ADR-0330 now freezes that
one-file network-only bind and pinned IPv4 probe. Its thin producer proves DNS
and completes the fetch, then v3 closes because the inventory rejects a
legitimate Cargo/libgit firmware pack-index hard link. V4 needs a fresh
canonical hard-link-row decision; no cache survives.

ADR-0331 now freezes one lexicographic owner plus explicit alias rows with
shared mode/size/hash/link count and outside-root rejection. Every non-inventory
preparation gate remains unchanged. Pushed v4 proves that inventory, then
closes because the inherited probe wrongly equates 162 active metadata packages
with all 169 lock entries. V5 needs structural package-ID/lock authentication.

ADR-0332 now freezes that structural validator and makes the active count/digest
result-only. All DNS/fetch/inventory/resource/no-compilation gates remain exact.

## Phase exit criteria

- A committed, regenerable result on an external target with the scoreboard
  discipline, plus the honest write-up. If the result is "the fragment was too
  small for this target," that is *recorded as the measured result* and drives
  the next P5.1 iteration — the phase does not silently move goalposts.
