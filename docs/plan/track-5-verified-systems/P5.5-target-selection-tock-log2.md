# P5.5 replacement target selection — Tock integer logarithms

## Decision

After ADR-0326 closes the Maestro owning-build route, select Tock's two public
integer-logarithm helpers as the replacement P5.5 external target:

- `kernel::utilities::math::log_base_two(u32) -> u32`; and
- `kernel::utilities::math::log_base_two_u64(u64) -> u32`.

The exact Tock revision is
[`ac5d597d22fbf3b03ef2169a577bac246ef65ffb`](https://github.com/tock/tock/commit/ac5d597d22fbf3b03ef2169a577bac246ef65ffb)
(tree `5243357a7034d3a5fa68487ea839a25e573a25ef`, 2026-07-21).
`kernel/src/utilities/math.rs` has SHA-256
`15df6044286702bb0c70e3c435c90acca6d04a3be0dad2446bf07790a8d88368`.

These are real kernel utilities rather than isolated benchmark helpers. The
32-bit function feeds Cortex-M MPU region sizing and SAM4L ADC clock selection;
the 64-bit function feeds the SAM4L watchdog scaler. The source contract makes
zero explicit and otherwise returns floor(log2(input)).

This note replaces target selection only. It admits no external artifact,
frontend feature, proof, performance number, or scoreboard row.

## Why this target follows the reviewer-aligned trajectory

The live reviewer plan asks for a second real axis and a rigorously bounded
generality claim, not another speed headline. Tock adds:

- a production embedded kernel rather than Glaurung or an Axeyum fixture;
- a current LLVM 22 compiler surface rather than Maestro's LLVM 20 surface;
- a leading-zero intrinsic with poison behavior, extending strict translation
  and definedness evidence rather than weakening it; and
- an MIT OR Apache-2.0 upstream declaration, while attribution and generated-
  artifact provenance remain explicit gates rather than inferred conclusions.

The target is deliberately small. It can establish authenticated external
capture and one exact semantic extension before storage-permission slices,
pointer memory, or full Tock kernel verification are considered.

## Non-crediting feasibility observation

A disposable full-tree materialization from the exact Git object built the
owning `kernel` release library with one codegen unit, linked-dead-code
retention, and textual LLVM emission. After a one-time locked cache fill, the
compile completed in 9.81 seconds and emitted one 2,651,673-byte, 34,904-line
module at SHA-256
`f9a1e1558d154b8238deae2f38f06ff251f6438ead8e109e4407b0e3998c76fd`.
This timing and module identity are feasibility observations only; they are not
expected values for the official two-root gate.

Both selected definitions are one-block scalar functions. Each compares the
input with zero, invokes width-matched `llvm.ctlz` with `is_zero_poison=true`,
maps the count to a zero-based high-bit index, and uses `select` so the poison
value is unobserved for the source-defined zero case. The 64-bit route also
uses a checked narrowing cast to the 32-bit return.

The current checked parser fails closed before solving. LLVM 22 prints a
call-result `range` attribute, which the parser does not yet type, followed by
the unsupported `llvm.ctlz` intrinsic. LLVM specifies that an out-of-range
return becomes poison and that `ctlz(x, true)` is poison for zero; neither may
be stripped as syntax:

- [LLVM `range` attribute](https://llvm.org/docs/LangRef.html#range-attribute);
- [LLVM `ctlz` intrinsic](https://llvm.org/docs/LangRef.html#llvm-ctlz-intrinsic).

## Preregistered obligation shape

For width `W` in `{32, 64}`, define an independent staged bit-search
specification `floor_log2_W`, using only existing BV comparisons, shifts,
extracts, and `ite`, not `ctlz` or the reflected function itself. The measured
run must prove:

1. `log2_W(0) = 0`;
2. `log2_W(x) = floor_log2_W(x)` for every `x : BVW`; and
3. for nonzero `x`, the returned index identifies the most-significant set bit
   (the selected bit is one and every higher bit is zero).

The 64-bit compiler result is compared after its explicit `i64 -> i32`
narrowing. Negative controls must independently alter the `31`/`63` index
constant, invert the zero arm, and corrupt one high-bit partition. Every SAT
control must replay against the reflected term and an independent native Tock-
source oracle.

## Immediate prerequisite and capture gates

ADR-0327 owns the prerequisite before any official external result:

1. parse a non-wrapping integer call-result `range(iN lo, hi)` as typed syntax;
2. preserve its LLVM rule by conjoining result-in-range with definedness;
3. add scalar `llvm.ctlz.iN(value, i1 constant)` with exact zero-poison
   semantics, rejecting vectors, nonconstant poison flags, width mismatch,
   wrapped/empty range forms, and unrelated count intrinsics;
4. canonically render/reparse both forms and extend the standing semantics gate
   with exhaustive small-width plus deterministic 32/64-bit differential rows;
5. keep this a reflection lowering over existing typed BV terms, not a new
   public Axeyum IR operator;
6. only after that prerequisite passes, preregister two independent Tock source
   and Cargo-target roots at identical virtual paths, a validated locked cache,
   no network during builds, raw complete-module equality, LLVM-22 extraction,
   exact selected symbol discovery, and atomic local-only output; and
7. require parser admission before a separate proof/scoreboard ADR.

No source edit, attribute stripping, LLVM-text normalization, single-root
capture, or post-observation range/poison relaxation is allowed.

## Rejected alternatives

- **Retry Maestro with a font cache or network.** Rejected by ADR-0326 after the
  registered isolated build observed that new input.
- **Tock storage permissions first.** Valuable later, but its enum/slice memory
  surface combines several unmeasured semantic extensions.
- **Tock CRC first.** Its slice memory and loops are a larger first external
  boundary; private table constructors are compile-time implementation details.
- **Strip `range` or rewrite `ctlz` in extracted LLVM.** Rejected: strict typed
  translation is the lead contribution, and both forms carry definedness.
- **Use the sparse reference checkout directly.** Rejected: official inputs
  must materialize the complete exact Git tree, not depend on sparse-worktree
  state.
