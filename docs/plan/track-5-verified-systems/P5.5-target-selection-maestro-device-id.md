# P5.5 target selection — Maestro device-number encoding

## Decision

Select the three scalar device-number functions in Maestro's
`kernel/src/device/id.rs` as the first P5.5 external target:

- `major(dev: u64) -> u32`;
- `minor(dev: u64) -> u32`; and
- `makedev(major: u32, minor: u32) -> u64`.

The selected upstream revision is
[`650a3f62c386d113b4cbbc11645d945d57620cbb`](https://github.com/maestro-os/maestro/commit/650a3f62c386d113b4cbbc11645d945d57620cbb)
(2026-07-14). The source blob SHA-256 is
`9da62c538cf9decb14dae949306ba5d73fb5ec5229109ffe9ddf7ee4742a40df`.
This is a real kernel utility rather than a benchmark fixture: the revision
uses the functions in device construction, initramfs parsing, filesystem
syscalls, and `stat` conversion. Maestro's own test exhausts only majors and
minors in `0..512`; the proposed Axeyum cell is universal over both `u32`
inputs and the entire `u64` device-number space.

This note completes target selection only. It contains no retained upstream
source or compiler output, reports no verification result, and authorizes no
scoreboard row.

## Exact identity and feasibility observation

The target pins `nightly-2025-05-10`:

- `rustc 1.89.0-nightly`, commit
  `dcecb99176edf2eec51613730937d21cdd5c8f6e`, LLVM 20.1.4;
- `cargo 1.89.0-nightly`, commit `7918c7eb5`;
- `kernel/Cargo.toml` SHA-256
  `6391c1b640b5b0aa565215a55f92947fcad63b2e913d63f490e4dd6f4561b0ec`;
- `kernel/Cargo.lock` SHA-256
  `23154f5d592ba682018737ee09d890754eeac0fcda7cd2a07d729b410a31a61d`;
  and
- `rust-toolchain.toml` SHA-256
  `f25aa01d7f83c31506438cdbc0ab86bf5799db7fc1018dd477b5856ff86e9d01`.

A disposable, memory-capped feasibility build ran the owning locked kernel
command with that exact toolchain and `--emit=llvm-ir`. It completed in 11.10 s
and emitted a 36,038,525-byte module with SHA-256
`2e7b51829eb1bb0fc732ea78bb443a7682836ae3afb1080762e7acc3fd87f187`.
The module contains all three exact mangled function definitions. Each is a
single basic block over scalar shifts, masks, disjoint ORs, casts, and LLVM
integer flags already represented by the checked scalar syntax.

That module is a feasibility observation, not a corpus artifact. It is not
committed, its timing is not a measured P5.5 result, and T5.5.2 must reproduce
it from a preregistered command before any semantic result is observed.

## Why this target wins the smallest-viable comparison

| Candidate | Smallest useful surface found | First unsupported demand | Decision |
|---|---|---|---|
| Maestro | the three device-ID scalar functions above | reproducible whole-kernel LLVM capture and exact extraction, not new operator semantics | **selected** |
| Tock | storage permissions, syscall classification, or integer-log helpers | enum/array/slice memory or the `ctlz` intrinsic | defer; materially larger first cell |
| Hubris | fixed strings and stateful protocol helpers | loops, memory, callbacks, and in places `unsafe` | defer |
| rust-sel4 | userspace helpers/tasks | allocation, strings, traits, or wider runtime context | defer |
| Asterinas OSTD | page-table code | general pointer provenance, aliasing, and wide memory | defer; highest fragment cost |

The choice follows P5.5's “pick by smallest viable” rule. It does not claim
that device-number packing represents kernel verification broadly. A later
target should deliberately add memory, state, or control-flow diversity after
this first external owning-build path is measured.

## Preregistered obligation shape for T5.5.3

T5.5.2 must freeze the precise terms before solving, but the intended
properties are already fixed by the source interface:

1. `major(makedev(M, m)) = M` for all `M, m : BV32`;
2. `minor(makedev(M, m)) = m` for all `M, m : BV32`; and
3. `makedev(major(D), minor(D)) = D` for all `D : BV64`.

Together these state that the encoding and decoding functions are mutual
inverses over the complete declared domains. The run must include independent
specification terms, exact compiler-body reflection, original-term model
replay, and source-level replay for every countermodel. Frozen negative-control
families must remove or move at least one high-major and one high-minor lane so
that small-range testing cannot satisfy the gate vacuously.

## T5.5.2 gap analysis and gates

The scalar semantics are already present. The missing work is the external
artifact boundary:

1. add a deterministic owning-Cargo LLVM capture command for the pinned
   manifest, package/target, target JSON, toolchain, environment, and complete
   ordered argv;
2. reproduce and authenticate the full module before extracting any function;
3. use the pinned LLVM toolchain to extract all three functions without a
   random-path `ModuleID` contaminating the canonical identity;
4. bind demangled and mangled names, source blob, manifests, toolchain, full
   module, extracted modules, and canonical checked syntax in one provenance
   manifest;
5. require two clean captures and root-independent regeneration, with no
   partial output on any mismatch;
6. prove parser admission before authorizing a measured run; a decline remains
   a result and must not cause source editing or a syntax-only shim;
7. provide a target-source replay route independent of reflected terms; and
8. resolve the distribution and attribution boundary before committing any
   Maestro-derived source or compiler fixture. The selected source declares
   GPL-3.0-or-later terms, while Axeyum's repository is MIT OR Apache-2.0. Until
   that review is recorded, T5.5.2 may commit only Axeyum-owned capture tooling,
   hashes, commands, and provenance metadata; it must regenerate third-party
   bytes locally and must not vendor them.

Items 1–7 are a bounded P5.1/T5.1.2–3 external-capture increment. They do not
authorize general whole-crate LLVM reflection, pointers, memory, or new public
operators. Item 8 is a release-engineering gate, not a claim about license
compatibility.

## Next action

Preregister T5.5.2 in a zero-row ADR. It must freeze the acquisition/build/
extraction commands, hashes and normalization rules, failure taxonomy,
licensing boundary, and byte-for-byte reproduction gates before retaining or
solving any extracted target function.
