## 1. Purpose and boundary

M2.1 and M2.2 establish what a Lean source declares and which configured
module artifacts its provider can load. They do not establish how CTest
dispatches the source, which generated environment reaches the runner, which
shell files are sourced, or which control branch selects a command. M2.3 must
close that gap for every selected case/provider projection without turning
lexical shell text into execution evidence.

The singular M2.3 rule is:

> A runner edge is configured only from exact generated bytes and an exact
> CTest registry; a shell command is executed only when its semantics are
> statically closed under the bound state or a separately registered trace
> observes it.

This plan freezes semantics before any M2.3 implementation or process. It does
not materialize a wrapper, configure Lean, run CTest, trace a test, resolve a
provider, or add U2/native/pair/parity credit. M2.3 input binding remains
downstream of an accepted M2.1 result and an accepted M2.2 authority/result.

M2.3 owns wrapper generation, CTest dispatch, shell sourcing, case hooks,
control markers, and the boundary command leaving a runner. It does not absorb:

- module/effective-import closure from M2.2;
- Lake manifests, setup, targets, facets, caches, or package effects from M2.4;
- compiler, linker, native runtime, FFI, or child-process effects from M2.5;
- editor request/transcript semantics from M2.6; or
- cross-provider merge/review from M2.7/M3.

An M2.3 edge into one of those surfaces must retain its downstream owner and
cannot close that downstream milestone.

## 4. Configuration and generated-wrapper semantics

### 4.1 Generation inputs

For each provider variant, bind the exact:

- source-copy identity and clean tracked tree;
- CMake executable, version, generator, platform/system variables, preset,
  cache, toolchain file, configure command, and ordered `-D` arguments;
- `STAGE`, `CMAKE_SOURCE_DIR`, `CMAKE_BINARY_DIR`,
  `CMAKE_CURRENT_SOURCE_DIR`, compilers, build type, `LEANC_OPTS`,
  `LEAN_TEST_VARS`, `LAKE_CI`, and all values contributing to `TEST_VARS`;
- environment allowlist and every path inherited into configuration;
- template bytes/mode, CMake input files, configure stdout/stderr/cache, and
  generated output path/bytes/mode; and
- test and benchmark wrapper identities separately.

`configure_file` is called without `@ONLY`, `COPYONLY`, explicit permissions,
or an explicit newline style. The authority must therefore bind the CMake
version and show the exact substitution result, source-permission inheritance,
and output newline bytes. A hand-written renderer may be an independent
comparator but cannot be the generation authority.

### 4.2 Ordered substitution map

`TEST_VARS` is shell source, not a neutral key/value object. The configured
wrapper must retain both the ordered pre-substitution fragments and final
bytes for:

1. configured `LEAN_TEST_VARS`;
2. `STAGE`;
3. `SRC_DIR`;
4. `TEST_DIR`;
5. `BUILD_DIR`;
6. `SCRIPT_DIR`;
7. the stage `bin` prefix plus inherited `PATH` expansion;
8. `LEANC_OPTS`; and
9. `CXX` plus its embedded compiler/options string.

The pinned CMake adds single quotes itself around most values but does not
perform a general shell-escaping transformation. Quotes, newlines, command
substitutions, backticks, semicolons, dollar expansions, glob characters, and
leading assignment-like text in any contributing value must be rejected or
shown to produce the preregistered exact shell parse. Merely decoding the
post-generation line into apparent assignments is insufficient.

The wrapper prepends its configured stage `bin` but preserves the launch
process's inherited `PATH`. M2.3 must bind both the literal configured fragment
and the effective runtime path. `bash`, `dirname`, `basename`, `mktemp`, `rm`,
`diff`, `grep`, `cat`, `perl`, `jq`, and every later external command resolve
under that effective state; tool names alone are not executable identities.

### 4.3 Source-tree mutation and collision rule

Pinned CMake writes:

- `tests/with_stage${STAGE}_test_env.sh`; and
- `tests/with_stage${STAGE}_bench_env.sh`

into the source tree, not the binary tree. Stage-specific names prevent a
stage-1/stage-2 collision, but two configurations of the same stage share and
overwrite the same path. A later configure can therefore change the wrapper
used by an earlier build tree's already-generated CTest registry.

M2.3 may configure only a fresh isolated source copy/overlay. Each provider
configuration gets a private source/output namespace unless an accepted
retained official build artifact already supplies immutable bytes. The helper
must capture a pre-tree inventory, permit only exact declared generated paths,
capture the post-configure inventory, then seal an immutable copy of the
wrapper before any other configure or test process. Every case projection
binds the wrapper hash observed at its launch boundary, not just the hash that
existed when CMake first configured.

Hard-link aliasing, symlinked output paths, cross-provider reuse, concurrent
same-stage configuration, undeclared source changes, or wrapper bytes changing
between discovery and launch fail closed.

## 5. Configured CTest registry

Generated wrapper bytes do not prove that a case uses them. For every bound
provider configuration, retain the complete configured CTest registry and its
raw files/JSON, including:

- test name, ordered command/argv, exact executable identity, working
  directory, configuration, labels, timeout/resource properties, fixtures,
  environment, `RUN_SERIAL`, disabled state, and all other properties;
- originating CMake file/span and selected profile/variant;
- exact configure-time glob result and lexicographic order;
- `.no_test`, `EXCEPT`, `run_test`/`run_bench` name exclusion, missing-runner
  warnings, and Lake include/exclude decisions;
- subdirectory traversal and every `CTestTestfile.cmake`/included registry
  byte identity; and
- selected-versus-registered projection after exact `CTEST_OPTIONS` filters.

CMake documents `file(GLOB)` results as lexicographically ordered and, without
`CONFIGURE_DEPENDS`, the target list is a configure-time snapshot. M2.3 must
not rescan the source tree later and call that the configured registry.

The existing U2 registration authority used pinned `tests/CMakeLists.txt` in a
deliberately minimal stage-1 harness. Its normalized wrapper digest
`5bc5d5f70087c932d0164bca3077fce7852d6f7342253dc5d9ef2407d228e61f`
and 3,723-case registry remain the case-denominator authority. They are not an
official provider configuration because the harness did not reproduce each
official compiler, cache, preset, platform, or build-tree input.

## 6. Shell state and typed runner graph

### 6.1 No blended shell graph

For each wrapper case retain four distinct layers:

1. **lexical candidates** — parsed tokens/expansions/branches, never edges;
2. **resolved static shell graph** — exact sources/commands whose expansion,
   branch, and resolution are closed under bound inputs;
3. **configured dispatch graph** — CTest and generated artifacts selecting the
   wrapper/runner/cwd; and
4. **observed trace graph** — exact sourced files and process executions seen
   in a registered instrumented run.

Each node/edge records its layer. A lexical `lean`, `lake`, compiler, plugin,
or filename occurrence cannot become an `executes`, `reads`, or `writes` edge.
An observed child process cannot retroactively prove an untraced branch or a
different provider variant.

### 6.2 Sourcing is shared state

The wrapper invokes `util.sh` with a slash-qualified path, then changes to the
registered runner directory and sources `$(basename "$TEST_SCRIPT")`. In Bash,
`source` of a name without a slash searches `PATH` when `sourcepath` is enabled
and only falls back to the current directory outside POSIX mode. M2.3 must bind:

- Bash executable/dynamic identity and `BASH_VERSION`;
- POSIX mode, `shopt -p sourcepath`, `SHELLOPTS`, `BASHOPTS`, `CDPATH`, `IFS`,
  locale, umask, limits, inherited traps/functions, and positional parameters;
- the `dirname`/`basename` executable identities and exact outputs;
- every searched source candidate and the file actually recorded in
  `BASH_SOURCE`; and
- pre/post cwd, environment, arrays, functions, options, traps, limits, and
  open trace descriptors at each source boundary.

The registered runner path is not proof that the same bytes were sourced. A
same-named file earlier in `PATH`, disabled `sourcepath`, POSIX-mode change,
runner symlink drift, or unexpected `cd` result must be observable and reject
unless explicitly preregistered.

`util.sh`, runners, and `.init.sh` files execute in the same shell. They may
change variables, arrays, functions, cwd, options, limits, traps, or terminate
the wrapper with `return`/`exit`. By contrast, `.before.sh` and `.after.sh` are
launched as child `bash --` processes. That isolation and their effective Bash
identity must remain explicit.

`tests/misc/run_test.sh` computes a real path but then sources the original
`$1`; each of its five registered `.sh` cases is arbitrary wrapper-shell code.
The 31 registered directory cases similarly source their directory runner with
no case argument. Static token scanning cannot close either form.

### 6.3 Shell control flow and failures

The graph must preserve `set -e`, subshell boundaries, pipelines, `&&`/`||`,
command substitutions, process substitutions, functions, arrays, globs,
loops, background jobs, `wait`, signals, traps, redirections, and status
propagation. In particular:

- `util.sh` creates a per-process temporary directory and installs an `EXIT`
  trap; both successful cleanup and residue are generated/write effects;
- `run_only` and `capture_only` execute commands in subshells and deliberately
  convert status into `EXIT`;
- `capture_only` redirects both streams before replaying the produced file;
- `check_out_file` gives `.out.ignored` precedence over `.out.expected`, then
  requires empty output when neither exists;
- `run_before`/`run_after` failure terminates through `fail`; and
- an early failure can make a later hook or branch reachable statically but
  not taken dynamically.

M2.3 records the runner boundary command and shell-generated/read/write edges.
The semantics and effects of Lean/Lake/compiler/server children transfer to
M2.4-M2.6.

## 7. Hooks, sidecars, and marker precedence

The pinned full-Lake cases currently bind 27 `.init.sh` files, six
`.before.sh` files, zero `.after.sh` files, 1,480 `.out.expected` files, and 60
`.out.ignored` files. One additional tracked init and all seven tracked
`.no_test` files belong to excluded candidates rather than registered cases.
These counts are assertions for M2.3.1 to rederive, not substitutes for exact
rows.

For `compile/run_test.sh`, M2.3 must reproduce these independent precedence
chains exactly:

```text
compile:
  .do_compile_test > .no_compile_test > .do_compile > .no_compile > default on
interpret:
  .do_interpret_test > .no_interpret_test > .do_interpret > .no_interpret > default on
```

The target population currently uses six `.no_interpret`, two
`.do_interpret`, and one `.no_compile` marker; the more-specific test markers
are absent. Absence must be recorded, not dropped. The init script is sourced
before marker checks. Each enabled compile/interpreter branch runs its own
before/after hook pair and output/exit check; a failed first branch prevents
the second under the shared `set -e` flow.

Every marker row records lexical path, resolved path, kind, existence/type,
mode, bytes/hash when present, precedence ordinal, selected branch, profile,
and source evidence. Directories, symlinks, dangling links, races, or file-type
changes at marker paths fail closed.

## 8. Static proof and observed-trace contract

### 8.1 Static closure

The later helper may statically close only a bounded Bash grammar with explicit
semantics and tests. It must parse rather than regex-match and retain every
unsupported construct as a residual. Expansion is allowed only when all input
values, option state, cwd, file tests, glob results, command resolutions, and
status-dependent branches are bound. `eval`, dynamic `source`, generated shell
text, unbounded loops, runtime-produced paths, inherited functions, or unknown
child status prevent static `executes` credit.

Static rows use `resolved-static` or `configured`; they never claim
`observed-runtime`. Branches proven false become `conditional-not-taken` only
with the exact condition and complete alternative registration.

### 8.2 Trace closure

Each remaining dynamic residual requires a separately preregistered trace for
the exact case/provider equivalence class or a reviewed decline. The trace
envelope must retain:

- exact uninstrumented command, instrumented command, cwd, allowlisted
  environment, Bash/tracer recursive identity, limits, timeout, process group,
  and network policy;
- xtrace on a dedicated descriptor with a quoted `PS4` carrying source,
  line, function, process, and subshell identity;
- source-stack observations sufficient to disambiguate basename lookup;
- platform-native process creation/exit observations for every external child
  and executable identity;
- raw ordered trace streams, truncation accounting, parser output, and joins;
- typed terminal state, complete child reaping, generated-file inventory, and
  completion-last immutable evidence; and
- a pointer to an independent uninstrumented official outcome when behavioral
  credit is later requested.

Tracing perturbs shell options, descriptors, scheduling, timing, and possibly
scripts that inspect their environment. A trace proves only the M2.3 edge it
observes; it is never the official result and cannot replace an uninstrumented
run. Child Bash instrumentation, if needed for hooks, must be explicit rather
than assumed to inherit. Secrets are forbidden from the input environment and
raw traces; redaction after capture is not an evidence strategy.

The exact residual population and process formula are deliberately deferred.
They can be computed only after accepted M2.2 provider projections and an
M2.3.1 static analysis freeze which determines equivalence classes and dynamic
residuals. No phrase such as “one trace per runner” may hide case-specific
init, hook, marker, source, platform, or configuration differences.

normalization is not identity.

## 12. Source-first sequence and process gate

1. **M2.3.0 plan:** this document freezes semantics from source, existing
   authorities, official documentation, and read-only artifact inspection. It
   runs no configuration, test, or trace process.
2. **M2.3.1 input/static authority:** only after accepted M2.1 and M2.2 results,
   bind provider configurations, complete source rows, synthetic controls,
   static grammar, exact residual/equivalence classes, process formula, limits,
   evidence root, and authorization digest. Observation and credit stay zero.
3. **M2.3.2 implementation:** implement configuration capture or retained-
   artifact ingestion, registry validation, shell parsing/abstract execution,
   trace parsing, immutable evidence validation, and semantic mutation tests.
   Commit and push before rendering an authorization command.
4. **M2.3.3 attempt:** only an exact later user authorization may run the
   frozen configuration/trace process program. Validate immutable evidence
   before a separate offline promotion.

Configuration and tracing are external processes even when no Lean test is
intended. Their exact count is not authorized by this plan. An accepted
official build artifact can replace a reproduction only when provenance,
configuration inputs, generated bytes, registry, and launch-time identity are
complete; “came from CI” is not an identity.
