# P3.2 — Alethe term/proof IR + emitter — KEYSTONE (critical path)

**Size:** M · **Depends on:** [P3.1 LRAT](P3.1-lrat.md) · **Keystone:** the whole
proof track downstream depends on this

## Goal
A new crate `axeyum-alethe`: the Alethe proof IR (terms, commands, steps), a
printer that emits valid `.alethe` text, an `Op`→Alethe-`Operator` map, and a
`polyeq` matcher. Alethe is the realistic primary proof target: Rust-checkable
(Carcara, fits no-C/C++), BV-shaped (matches axeyum's lowering and miter), and the
on-ramp to Lean reconstruction.

## Design (mirror Carcara's `ast/`)
- Three command forms: `(assume id term)`, `(step id (cl term*) :rule name
  [:premises (..)] [:args (..)] [:discharge (..)])`, `(anchor :step id [:args ..])`.
- `(cl)` = empty clause = false; rules are **strings**; unknown rules can be
  `hole` (lets axeyum emit partial proofs and fill rules incrementally).
- Hash-consed terms with pointer equality + a separate **`polyeq`** (equality
  modulo reorder/AC/alpha). axeyum already has the interned `TermId` arena; add
  the polyeq matcher.
- The `Operator` set already carries SMT-LIB BV names (`bvadd`, `bvult`, `concat`,
  `extract`, `@bbterm`, `@bit_of`) — a 1:1 map to axeyum's `Op`.

## Phase exit criteria
- `axeyum-alethe` emits well-formed `.alethe` text that **Carcara parses**
  (round-trip / parse test in CI, Carcara is Rust so it can be a dev-dependency or
  vendored), with a working `Op`→`Operator` map and a `polyeq` matcher. No proof
  *content* yet — that's [P3.3](P3.3-alethe-qfbv.md) — but the format pipeline is
  in place.
