## One-paragraph thesis

axeyum's strings are a **bounded** ≤16-byte bit-vector encoding (the HAMPI/Kaluza
approach): sound, but it can only decide what fits in 16 bytes and a Thompson-NFA
regex fragment. Z3 and cvc5 reason about **unbounded** strings symbolically. We
close the gap by building a **word-level, length-aware DPLL(T) string theory
solver** in the cvc5 lineage — congruence-closure + string derivation rules,
combined Nelson-Oppen-style with the LIA solver over shared `len(x)` terms —
augmented with **symbolic-derivative regex** (the Z3 `seq` technique) and, later,
an **automata/stabilization fallback arm** (the Z3-Noodler technique, but on a
**pure-Rust** automata substrate, since the no-C/C++ rule forbids cvc5/Noodler's
C++ MATA/libpoly shortcuts). The bounded encoder is retained as a fast pre-check
for small instances. Everything stays `unknown`-first and **DISAGREE=0**.

## The scoping fact that governs everything

**Decidability of quantifier-free word equations + length constraints is a
long-standing OPEN problem** (≈ since 1968; Ganesh et al. 2013). Concatenation +
LIA-over-length + string⇄int conversion is outright **undecidable** (Ganesh &
Berzish 2016). So **no string solver is complete** — the SMT-COMP winners are all
*refutation-sound and solution-sound but deliberately incomplete and
non-terminating in general*. axeyum adopts that incompleteness **honestly**: we
target the well-understood decidable fragments (straight-line, acyclic,
chain-free, regex+LIA) and return `unknown` outside them, **never** a wrong
verdict. This matches our "untrusted fast search, trusted small checking" and
"`unknown` is first-class" stances perfectly.

## Why this, why now

- **Measured gap** ([gap analysis Gap 10](../gap-analysis-z3-cvc5-2026-06-22.md)):
  strings are `experimental` — no IR sort, no variable-length results, `str.len`
  unsat can be `unknown` due to the BV+LIA combination gap.
- **It reuses infrastructure we're already building:** the string theory solver is
  "EUF on the e-graph ([P1.4](../track-1-engine/P1.4-egraph.md)) + derivation rules
  + a Nelson-Oppen link to the LIA online solver over `len` terms." Closing the
  BV/String+LIA gap is [P1.6](../track-1-engine/P1.6-theory-combination.md); the
  rest is string-specific reasoning on top.
- **The differential fuzz already passes:** an adversarial strings fuzz vs Z3 is
  DISAGREE=0 over 371 instances — the bounded encoder is sound; we are extending
  reach, not fixing correctness.

## Sequencing at a glance

```
Phase A (IR sort + String/LIA combination)   [prereq refactor; closes str.len unsat]
        │
        ├──> Phase B (word-equation core)     [the decidable-fragment engine]
        │        │
        │        ├──> Phase C (derivative regex)
        │        ├──> Phase D (extended functions: lazy reduction + CDS)
        │        └──> Phase E (models + automata fallback arm)
```

Phase A is the enabling refactor (the `Parsed = Term | Str` / first-class sort
decision). Phase B is the core engine that makes strings unbounded. C/D/E layer on
top and can proceed partly in parallel once B's normal-form invariant exists.

## Standing discipline (every phase)

- **`unknown` first-class** outside the decidable fragments / past budget.
- **DISAGREE=0** — every `sat` model replays against the original term (the
  `str_differential_fuzz` vs Z3 gates every change); every `unsat` route gets a
  checker / trust-ledger entry. *Cautionary tale:* both cvc5 and Z3str4 shipped
  **soundness bugs** flagged at SMT-COMP 2021 — string solvers are subtle; test
  harder.
- **No C/C++** — the automata fallback builds on pure-Rust substrates
  (`regex-automata`, `aws-smt-strings`, `smt-str`), **not** MATA/libpoly.
- **SMT-LIB Unicode Strings** alphabet: code points `0x00000–0x2FFFF` (Planes 0–2),
  with a total order (load-bearing for the model-construction argument and our
  determinism promise).
