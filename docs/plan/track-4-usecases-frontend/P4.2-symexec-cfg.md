# P4.2 — Symbolic-execution CFG frontend (angr/unicorn-class)

**Size:** XL · **Depends on:** [P4.1 warm lazy memory](P4.1-warm-lazy-memory.md) ·
**Capstone of the symbolic-execution use case**

## Goal
Build the binary/IR frontend that turns axeyum's solver-side symbolic-execution
primitives (`SymbolicExecutor`, BMC, k-induction, path-condition optimization)
into an angr/unicorn-class tool: lift a program to an IR, build a CFG, explore
paths with a real memory model, and cross-check against concrete emulation.

## Current slice
The first frontend primitives are in place. `SymbolicMemory` declares or wraps
an SMT array memory state, builds `select`/`store`, and routes load-equality
branch/assume queries through `SymbolicExecutor`'s memory-aware feasibility
path. It also exposes `SymbolicMemoryWrite` plus conservative write-log
normalization/read helpers: frontends can drop same-index shadowed writes and
emit one deterministic read-over-write `ite` guard per visible write while
preserving last-writer-wins for symbolic aliases. `SymbolicExecutor::explore_cfg`
is the reusable DFS harness: a frontend
supplies CFG states and `CfgStep`s, while axeyum handles branch feasibility,
solver scopes, infeasible pruning, unknown-safe traversal, and model-witnessed
target reporting. `explore_cfg_checked` adds the concrete-emulation seam: for
each symbolic target, the frontend extracts a concrete witness from the model and
checks it with an independent emulator before the target is counted as verified.
`TinyBvProgram` is the first reusable small-target frontend: it validates a
fixed-width BV register program, declares symbolic inputs, lifts instructions to
`CfgStep`s, extracts concrete model witnesses, and independently replays those
witnesses in a concrete emulator. It also has bounded program-counter
reachability/safety wrappers: reachable/unsafe reports carry concrete-replayed
witnesses; unreachable/safe is reported only when the bounded search is
exhaustive and diagnostic-clean. The toy target now includes `Load`/`Store`
instructions over a zero-initialized SMT array memory and concrete-replays them
with the same zero-default map. Concrete replay also returns an executable trace
of PCs/instructions, register snapshots, final registers, explicit memory cells,
and outcome. `TinyBvProgram::from_assembly` adds a small imported text format for
the toy target (`const`, arithmetic, `load`/`store`, `beq`, `win`/`lose`), with
labels for branch targets, register-vs-register equality branches
(`beq rA rB ...`), line-numbered parse errors, a public label-to-PC map, a
public PC-to-source-line map for imported instructions, deterministic
PC-to-label lookup, typed static CFG edges, source/label-aware basic blocks,
deterministic Graphviz DOT export for the basic-block CFG, trace-highlighted
and block-/edge-coverage-highlighted DOT overlays, compressed block trace paths,
taken CFG edge reports, source-aware concrete trace rows, consolidated witness
replay reports, replay-checked test-case generation reports, block-coverage and
edge-coverage test-suite reports, and label-based
reachability/safety query wrappers over the existing checked PC queries. These
are frontend helpers over the current one-shot dispatcher, not the final warm
lazy array engine from P4.1.

2026-06-27 update: `SymbolicExecutor::branch` now auto-promotes branch
feasibility checks to the memory/theory-aware dispatcher when the current path or
branch condition contains arrays or uninterpreted-function applications, and
`explore_cfg` does the same for branch, assume, target-status, and target-model
queries even when `CfgExploreConfig::memory_aware` is left at its default
`false`. The explicit `*_with_memory` methods remain available, and new
`assume_auto`/`status_auto`/`model_auto` helpers expose the same route selection
for direct callers. This is an ergonomics/product slice over the current
one-shot full dispatcher; it is not the retained-lemma lazy-array/UF engine from
P4.1.

2026-06-27 warm ROW update: `assume_auto` and `branch` now first ask the
incremental solver to simplify syntactic same-index read-over-write terms for
warm encoding while retaining the original memory term for replay. One-shot
assumption cores are reported in terms of the original branch assumption.
Store/read-back constraints such as `select(store(mem, i, v), i) = target` can
therefore use ordinary warm `status`/`model` and branch feasibility checks;
general symbolic-memory and UF constraints still auto-route to the memory-aware
dispatcher.

## Why / leverage
The solver side is built (path exploration, test-suite enumeration, optimization,
reachability, certified safety). What's missing is the *frontend*: a program
representation, a CFG-shaped explorer, and a memory model — the difference between
"a solver with symexec primitives" and "a symbolic-execution engine."

## Phase exit criteria
- A working CFG-shaped symbolic executor over a small target: lifts, explores with
  pruning, models memory warmly, generates test inputs, answers reachability/safety
  with witnessed traces or certified proofs, and cross-checks against concrete
  emulation. (Scope is intentionally a *small* target first; broadening ISAs is
  follow-on.)
