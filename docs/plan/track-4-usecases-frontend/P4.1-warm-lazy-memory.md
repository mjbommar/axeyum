# P4.1 — Warm lazy arrays / symbolic memory

**Size:** L · **Depends on:** [P1.4 e-graph](../track-1-engine/P1.4-egraph.md),
[P1.5 CDCL(T)](../track-1-engine/P1.5-cdcl-t-loop.md),
[P2.2 lazy arrays](../track-2-theories/P2.2-arrays-lazy.md) · the deferred half of
ADR-0030

## Goal
Make the incremental engine handle arrays **warmly** so memory-bearing symbolic
execution / BMC / k-induction stop re-eliminating arrays eagerly on every
`check`/depth. Today the warm path **refuses arrays** and `check_with_memory`
re-eliminates per call (one-shot), which is the perf wall for memory workloads.

## Why / leverage
The already-shipped `bounded_model_check_with_memory` and `check_with_memory`
re-solve from scratch each depth. With lazy arrays on the warm CDCL(T) engine
(P2.2), learned clauses and array axioms persist across depths — the difference
between toy and real symbolic-execution-over-memory.

Interim U7 mitigation (2026-06-27): `SymbolicMemory` now has a frontend
write-log helper that drops later-shadowed same-index writes and emits a compact
read-over-write `ite` chain for a read. This reduces consumer-side store-log
blowup for simple memory logs, but it is not a replacement for retained
lazy-array/UF theory clauses.
