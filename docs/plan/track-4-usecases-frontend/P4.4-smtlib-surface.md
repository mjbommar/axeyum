# P4.4 — SMT-LIB command/API conformance

**Size:** M · **Depends on:** — (independent; `get-proof` ties to
[Track 3](../track-3-proof-lean/README.md))

## Goal
Turn the existing parser, incremental command stream, and single-query Rust
helpers into a precisely scoped SMT-LIB 2 protocol surface. The immediate gap
is no longer “add every helper”: it is preserving command-point ordering,
emitting a textual response for every output command, enforcing option and
session lifecycle semantics, and exposing already-implemented direct engines
without pretending a parser no-op is a command implementation.

## Why / leverage
Mechanical but broad — it is what lets external tools and benchmark drivers use
axeyum without special-casing. The checked
[SMT-LIB/API conformance matrix](../generated/smtlib-api-conformance.md) now
provides the source of truth: 29 high-value rows distinguish syntax acceptance,
execution, output representation, assurance, exact tests, and residual gaps.
Its central result is that many useful helpers already exist, but **zero rows
provide an ordered interactive textual session**.

## Phase exit criteria
- A generated matrix covers every admitted high-value command and records parse,
  execution, output, assurance, scope, exact tests, and residual behavior.
- One ordered session runner covers state changes and output commands without
  re-solving against the wrong query or reading final global state for an
  earlier request.
- Full `reset`, option updates, `exit`, output channels, and error behavior have
  transcript fixtures. Unsupported commands fail visibly.
- Proof, interpolation, Horn, and abduction text surfaces expose only their
  measured direct-API fragments and preserve verify-before-return boundaries.
- Parametric sorts and recursive definitions are either implemented with their
  semantic obligations or retained as explicit, tested rejections.
