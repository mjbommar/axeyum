# P4.3 — Optimization: OMT lexicographic/Pareto + MILP hardening

**Size:** M · **Depends on:** — (independent; builds on existing `optimize.rs`,
`maxsat.rs`, MILP) · **Use case:** constrained program optimization

## Goal
Grow axeyum's optimization from single-objective (`maximize_bv`/`minimize_lia`,
MaxSAT, MILP — currently `experimental`) to Z3-`opt`-class: **lexicographic** and
**Pareto** multi-objective optimization, with hardened MILP and per-step certified
optima.

## Why / leverage
"Constrained program optimization" is a named first-class use case. The pieces
exist (per-step optimum certified by the underlying procedure; path-condition
optimization landed in `SymbolicExecutor`); what's missing is multi-objective
orchestration and robustness.

## Current slice
Lexicographic, box, and Pareto OMT modes exist for the main BV/LIA surface, and
the consumer-facing counterexample path now has a shared minimization helper:
`minimize_model` / `minimize_model_with_config` and `Solver::minimize_model`
lexicographically minimize a replay-checked satisfying model over caller-selected
Bool, unsigned-BV (`w <= 127`), and Int symbols. `produce_evidence_minimized`
and `prove_minimized` surface the same strict minimizer through the evidence and
proof front doors, so property and verification frontends can request a
deterministic "small failing input" without reimplementing optimizer loops
client-side. The metadata-aware `minimize_model_objectives` /
`Solver::minimize_model_objectives`, `produce_evidence_minimized_with_objectives`,
and `prove_minimized_with_objectives` variants add signed two's-complement BV
objective order for SDK/frontends that know a BV symbol represents a signed
Rust integer. Wider BV, Real, array, datatype, and uninterpreted-carrier
objectives remain explicit unsupported/unknown cases rather than silently
producing a non-minimal model.

2026-06-27 SDK slice: `axeyum-property` v0 is the first typed consumer of this
proof/minimization surface. It provides `Bool`, `Bv<W>`, and `Int` handles over
`TermArena`, records declared scalar symbols in deterministic objective order,
delegates to `prove` / `prove_minimized`, lifts scalar model values through typed
handles, and exposes reusable typed unsigned BV overflow predicates for EVM and
Rust-verifier frontends. It is deliberately a thin SDK; no solver logic lives in
the crate.

The first counterexample-to-test layer is also in place: a disproving model can
be extracted as a deterministic `Counterexample` over SDK-declared inputs and
rendered as Rust `let` bindings or a `#[test]` skeleton for native Bool, Int, and
BV<=128 values. Direct named and tuple symbolic bundles can render aggregate
Rust initializer statements over those scalar bindings, explicit nested
aggregate fields can be composed through caller-supplied expressions, and test
skeletons can insert caller-owned imports/module prelude plus setup snippets
before helper-rendered Boolean or Result-returning replay adapters. The SDK
does not invent frontend/domain replay semantics.

The first `Symbolic` trait slice is in place as well: `Property::symbolic::<T>`
and `Property::concrete::<T>` support Bool, unsigned fixed-width Rust integers
as BV terms, `i128` as Int, unit, and 2-/3-tuples with deterministic field names.
`Property::symbolic_struct` now covers macro-free named-field bundles for
struct-shaped inputs, and `#[derive(axeyum_property::Symbolic)]` now lowers
named, tuple, and unit structs to that surface. Signed fixed-width Rust integers
`i8`/`i16`/`i32`/`i64` now use two's-complement BV terms with signed model
lifting, signed-order minimized counterexamples, and signed Rust
counterexample literals.

The first app-level SDK measurement gate is also committed:
`crates/axeyum-property/tests/corpus.rs` and
`docs/consumer-track/property/SCOREBOARD.md` cover eight graduated property
workflows with 2 proved, 6 disproved, 0 unknown, 0 mismatches / DISAGREE, and
1/1 Lean-required case available. The gate is now generated from
`tests/support/corpus_cases.rs`, with a committed machine-readable
`docs/consumer-track/property/corpus.json`; external proptest/Kani-style
baselines remain open.
