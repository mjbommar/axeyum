# P1.8 — Strategy & tactics (per-instance engine selection)

**Size:** M · **Depends on:** — (more valuable once P1.7 PBLS + P2.4 native LIA
exist to choose between)

## Goal
A small tactic/strategy layer so axeyum picks an engine/pipeline **per instance**
(by logic and by cheap formula/resource probes) instead of one fixed pipeline —
the mechanism Z3 uses to be fast across diverse inputs.

## Why / leverage
Z3's `qfbv_tactic` is a probe-guided `cond` tree (eq-only BV → `bv1_blaster`;
normal BV → bit-blast+aig+sat; else fallback), with memory/shape probes making the
choice dynamic. axeyum has `axeyum-solver`'s `auto.rs` routing and ADR-0019
swappable strategies; this phase grows that into a real combinator + probe layer.

## Phase exit criteria
- A `cond`-tree strategy that, on the committed slice, matches or beats the best
  single fixed pipeline (measured), choosing per instance from {bit-blast, PBLS,
  native-LIA, eq-only fast path}.
- Logic detection routes SMT-LIB `(set-logic …)` to the appropriate script.
