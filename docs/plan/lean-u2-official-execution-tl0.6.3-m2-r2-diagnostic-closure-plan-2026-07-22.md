## 1. Decision boundary

R2 is a **zero-process, zero-credit diagnostic closure**. It may append a
bounded artifact projection and a diagnostic completion to the existing R1
evidence root. It may not launch CTest or any selected case, create a second
attempt, overwrite an R1 byte, create the original `post.json`,
`projection.json`, or `completion.json`, or turn R1's 64 diagnostic JUnit rows
into official outcome credit.

This order preserves the parent plan's fail-closed rule: R1 remains an invalid
attempt under its preregistered artifact contract. R2 exists to retain the
outcome-bearing generated bytes, prove the exact family-policy defect, and
measure a sustainable future store before any source-first retry plan.

## 4. Append-only diagnostic namespace

R2 may add only:

- `diagnostic/generated/<original-relative-path>` for the exact 67 retained
  payloads;
- `diagnostic/post.json`, binding the 124-row full manifest, the 67 retained
  descriptors, the 56 manifest-only descriptors, the already-retained wrapper,
  original-source equality, the R1 terminal/JUnit/case identities, and zero
  credit; and
- `diagnostic/completion.json` last, binding the original 83-file manifest plus
  every new diagnostic dependency.

All new files are regular, mode `0444`, installed no-replace, fsynced, and
revalidated after completion. Existing files, including raw bytes and case
records, must remain byte-identical. A conflict, symlink, unexpected path,
mutable file, original-source drift, generated-manifest drift, missing payload,
or completion-before-dependency stops without repair.

The diagnostic completion schema and output must say:

- `process_attempts_added = 0`;
- `official_cases/outcomes/passes/failures = 0`;
- `local_physical_shards_completed = 0`;
- `r1_attempt_state = invalid-post-artifact-closure`; and
- every parent/provider/Axeyum/pair/performance/population/axis/gate/parity
  counter remains zero.

## 5. Required offline gates

Before appending a byte, R2 tests must:

1. validate the R1 authority seal, physical digest, exact embedded invocation
   streams, and every existing 83-file record/raw linkage;
2. reject any mutation to the terminal, JUnit, case order/outcome, R1 credits,
   original evidence manifest, or absent original completion paths;
3. prove all 64 cases map to the exact family-specific artifact sets and that
   the three docparse passes require only `.out.produced`;
4. reconstruct the exact 124/67/56/1 split and all four frozen digests from the
   retained work root;
5. reject original file mutation, extra generated paths, missing retained
   payloads, changed metadata-only rows, symlinks, unsafe mappings, or work-root
   substitution;
6. exercise append-only installation and completion-last recovery in a copied
   synthetic store, including overwrite and interruption mutations;
7. prove the original 83-file root still validates before and after the append;
8. run the complete Lean/parity documentation suite and every generator/check;
   and
9. expose no process-launch command.

The plan, implementation, and tests must be committed and pushed in separate
checkpoints before the live evidence root is appended.

## 6. Thread-failure handoff, not correction

R1 observed `failed to create thread` in two compile-bench and 32 docparse
rows. R2 does not reinterpret those rows or change execution. It records two
pinned source facts for a later retry plan:

- `tests/docparse/run_test.sh` invokes `lean ... --run` directly and does not
  expand `TEST_LEAN_ARGS`; and
- pinned runtime `src/runtime/thread.cpp` reads `LEAN_STACK_SIZE_KB` in
  `lean_run_main`, while `src/Lean/Shell.lean` documents `-s/--tstack` in KiB.

A future process plan must independently preregister and control any universal
stack environment, demonstrate that it reaches direct docparse and generated
executables, preserve explicit worker/resource semantics, and use a new attempt
ID/root. R2 does none of those things.

## 7. Non-claims and stop condition

R2 does not validate R1 under the original plan, create official outcomes,
complete shard `0001`, reproduce an official provider, observe Axeyum, form a
pair, measure comparable performance, or advance any parity population, axis,
or gate. The terminal Lean-parity claim remains false.

If the exact work root or existing evidence differs, R2 stops permanently and
publishes the mismatch; it does not rerun Lean or reconstruct missing bytes.
After a valid diagnostic completion is committed and pushed, a separate
source-first retry plan may use the measured storage split and pinned runtime
stack mechanism.
